# 設計思想（基本原則）

## 1. エージェントの出力は「叩き台（ドラフト）」であり、最終確定は人間が行う

本アプリのユーザーであるサービスエンジニア（特にRSEは最も知識・経験に長けたシニア）は、
エージェントより高い専門判断力を持つ。一方で生成AI（LLM）の出力は確率的であり、同じ入力でも
項目名・順序・粒度がゆらぐ。内容は概ね妥当でも、現場の最終判断をそのまま委ねるべきではない。

したがって本アプリの基本原則は次の通り：

> **エージェントが生成する成果物はすべて「ドラフト」である。人間（FSE / RSE）が内容を
> 確認・編集・追記・削除し、最終確定の権限と責任を持つ。**

これは human-in-the-loop の正しい形であり、AIの限界を人間の専門性で補完するアプリ設計である。

## 2. すべての出力は編集可能にする（同一UX）

上記原則の具体化として、エージェントが提示するすべての構造化成果物は、画面上で編集できる：

- 各項目（テキスト）を編集できる
- 項目を追加・削除できる
- 編集状態はローカルに保存され、リロードしても保持される
- 「リリース」「発行」「ダウンロード」など下流に渡すアクションは、必ず**編集後の内容**を使う

UXは全成果物で統一する（編集可能なフォーム／表＋必要に応じてリリース/発行ボタン）。

### 対象となる成果物

| 成果物 | マーカー | 編集UX |
|---|---|---|
| 顧客ヒアリング項目（局面①） | `[[hearing]]` | 項目/観点/メモ編集・行追加削除・チェック |
| トリアージ表（推定原因・類似傾向） | `[[triage]]` | 各区分の内容を編集 |
| 作業チェックリスト（FSE実施） | `[[steps]]` | 項目/詳細/注意編集・行追加削除・チェック |
| FSE→RSE 引き継ぎ要約 | `[[handoff_draft]]` | 全欄編集・発行 |
| RSE→FSE ネクストアクション | `[[rse_actions]]` | 項目編集・行追加削除・リリース |
| HQ→営業所 派遣ブリーフィング | `[[dispatch_briefing]]` | 全欄編集・リリース |
| SR報告書ドラフト | `[[report]]` | 本文編集・Wordダウンロード |

## 3. ペルソナ判別は従業員IDで行う

ユーザーはチャット開始時に従業員IDを入力する。エージェントはIDの接頭辞でFSE/RSEを判別する。

- `RSE` で始まるID（例: `RSE0001`）→ リモートサポートエンジニア（HQトリアージ）
- `FSE` で始まるID（例: `FSE0001`）→ フィールドサポートエンジニア（現場対応）

判別したペルソナは、その会話の以降の応答すべてに適用する。IDが未入力の場合は、まず入力を促す。
（本モックアップの簡易ルール。将来は認証基盤の従業員マスタと連携する想定。）
